# Threat Model — Merge Mode and AUTO_APPROVE_MEDIUM

This document captures the threats introduced when Béa runs in continuous self-improvement mode with automatic promotion (`BEA_IMPROVEMENT_MODE=merge` or `BEA_AUTO_APPROVE_MEDIUM=1`).

## Scope

- **Merge mode**: the self-improvement daemon applies patches and runs tests automatically.
- **AUTO_APPROVE_MEDIUM**: MEDIUM-risk actions are executed without explicit human approval.
- **Human-in-the-loop (HITL)**: CRITICAL and HIGH-risk actions still require operator approval.

## Threat actors

| Actor | Motivation | Capability |
|---|---|---|
| Compromised LLM / jailbreak | Make Béa modify itself in unintended ways | Inject instructions through mission prompts, tool outputs, or memory entries |
| Malicious tool / MCP server | Escalate privileges or exfiltrate data | Return crafted outputs that exploit deserialization, eval, or shell execution |
| Compromised operator token | Act as an authorized human | Use `BEA_OPERATOR_APPROVE_IMPROVEMENT` or Telegram bot to approve harmful patches |
| Bug in ranking/scoring logic | Auto-promote a bad patch | Exploit weaknesses in the harness or benchmark suite to inflate scores |

## Threats and mitigations

### T1 — Prompt injection hijacks self-improvement

**Description**: an attacker embeds instructions in a mission prompt, web research result, or memory entry that cause Béa to generate a patch bypassing the gate.

**Mitigations**:
- Gate is implemented in `kernel/improvement/gate.py`, which cannot be modified by a patch without going through the gate itself.
- `kernel/improvement/gate.py` requires a 24h cooldown and limits consecutive failures to 3.
- Protected paths (`core/self_improvement/protected_paths.py`) block merges touching `kernel/`, `core/security/`, `api/auth.py`, CI, or payment code.
- All auto-applied patches must be signed (TODO: See Task 4 Step 4 of the consolidation roadmap).

### T2 — Secrets exposure via generated patches

**Description**: a patch generated by the improvement loop embeds `.env` values, API keys, or database URIs.

**Mitigations**:
- Pre-commit hooks (`detect-secrets`, `gitleaks`) block commits matching secret patterns.
- `core/self_improvement/sandbox_executor.py` runs with `--network=none` and scrubbed environment.
- `.env`, `.env.local`, `.env.*.local`, and `.env.bak*` are in `.gitignore` and never committed.
- Promotion pipeline strips known secret patterns before diff submission.

### T3 — Destructive action approved through MEDIUM auto-approve

**Description**: `AUTO_APPROVE_MEDIUM=1` allows actions such as file writes, shell commands, or tool invocations without human confirmation. A confused agent could overwrite critical code or delete data.

**Mitigations**:
- Approval bridge (`core/autonomy/approval_bridge.py`) classifies every action.
- MEDIUM-risk actions are auto-approved only when `BEA_AUTO_APPROVE_MEDIUM=1` is explicitly set.
- Destructive actions (DELETE, INFRA, DEPLOY) are classified HIGH or CRITICAL and never auto-approved.
- Rollback instructions are generated for every auto-approved patch; the harness validates rollback before promotion.

### T4 — Payment or billing manipulation

**Description**: a patch modifies Stripe/Railway/Reward integrations to redirect funds, change pricing, or disable billing checks.

**Mitigations**:
- Payment code paths are in protected paths; auto-merge is not allowed.
- Any change to `core/finance/`, `business/revenue/`, or `tools/integrations/stripe_tool.py` requires human review.
- Stripe keys are never committed; production requires `BEA_PRODUCTION=true` and valid secrets.

### T5 — Harness manipulation creates a false sense of safety

**Description**: a patch modifies the benchmark suite or test expectations to make a bad change pass.

**Mitigations**:
- `core/self_improvement/eval_harness.py` uses frozen tasks and deterministic scoring.
- `scripts/validate_local.py` runs security hardening tests, ruff, mypy delta, bandit, pip-audit, and silent-swallow baseline.
- Promotion pipeline runs the full regression suite (`pytest tests/ -q`) in a clean sandbox network.

## Review rules for critical actions

The following actions always require an explicit human decision, regardless of `AUTO_APPROVE_MEDIUM`:

- Modifying `kernel/`, `core/security/`, `api/auth.py`, `.github/workflows/`, `deploy/`.
- Accessing or rotating secrets (`scripts/rotate_secrets.sh`, `.env`).
- Deploying to production (Railway, VPS, Docker registry).
- Billing or payout changes.
- Disabling or weakening gate constants (`MAX_PER_RUN`, `COOLDOWN_HOURS`, `MAX_FAILURES`).

## Operational checklist

Before enabling `BEA_IMPROVEMENT_MODE=merge` or `BEA_AUTO_APPROVE_MEDIUM=1`:

- [ ] `BEA_API_TOKEN`, `BEA_ADMIN_PASSWORD`, and DB passwords are strong and not defaults.
- [ ] Telegram `TELEGRAM_ALLOWED_USERS` is limited to the operator.
- [ ] `scripts/verify_prod.sh` passes.
- [ ] Patch signing is configured (future requirement once implemented).
- [ ] A recent backup exists (`scripts/backup_db.sh`).
